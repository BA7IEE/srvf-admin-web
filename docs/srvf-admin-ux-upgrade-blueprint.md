# SRVF Admin 产品化升级蓝图（2026-07-10）

> **性质**：体验层升级路线图。回应主诉「后台太偏技术化，不说普通人了，连我很多都搞不懂怎么用」。
> **方法**：全站 28 个可见页 + 5 个隐藏可达页逐页清单化（路由/文案/表单字段/操作按钮全量取证）+ 后端 handoff（`../srvf-nest-api/docs/handoff/admin-web.md`）任务→端点图对读 + 6 条真实任务代码级走查 + 大厂 B 端方法论对标。
> **基线**：main `5b99816`（P1 srvf-kit #74–#78 与全局搜索 #79/#80 之后）· 后端 v0.37.0。
> **定位**：三部曲之三。`srvf-admin-vnext-blueprint.md` 管**功能覆盖**（已完成，105/105 契约全绿）；`srvf-ux-audit-2026-07.md` 管**表层修复**（A~D 档已落地）；本文管**产品化**——把「给后端契约做的 UI」改造成「没读过文档的干部也会用的工作系统」。实施前逐项过 §8 待拍板清单。

## 0. 一句话结论

**功能是全的，文案修过一轮，但产品仍然要求用户先学会后端的世界观才能干活。** 上一轮审计修的是「皮肤」（文案/搜索/按钮色）；这次要修的是「骨相」：

1. **概念层**——操作界面前必须先理解 会籍/任职/督导/角色绑定/Scope/两道门/两级审 这套模型，而界面从不解释；
2. **流程层**——「办一场活动」「让新人入队」「给人授权」都是跨多页多弹窗的长链条，界面没有任何流程引导，全靠脑内地图；
3. **信任层**——通知铃铛至今是脚手架假数据（「小铭 评论了你」），登录框预填 demo 账号，一眼「未完工」。

大厂后台（企业微信/钉钉/飞书管理后台、Ant Design Pro 系）的共同答案不是更漂亮，而是**换组织原则**：以「用户要完成的任务」为中心组织一切，登录即待办，流程画在界面上，配置藏到后面，界面只说用户的语言。本文给出对应的 IA v3、工作台 v2、流程四件套、术语系统与分期路线（U0~U4）。

## 1. 为什么修过一轮还是难用——三层病根模型

| 层                      | 内容                                                              | 状态                       |
| ----------------------- | ----------------------------------------------------------------- | -------------------------- |
| L1 表层（皮肤）         | 文案直出枚举、缺搜索、按钮一片红、权限空态                        | ✅ 2026-07 审计 A~D 档已修 |
| L2 概念层（世界观）     | 界面词汇=后端模型词汇；操作前提是理解数据模型；配置面与业务面混陈 | ⛔ 未动——本文 §4.3/§4.6    |
| L3 流程层（任务地图）   | 高频任务跨 N 页 N 弹窗，无向导、无流程图、无「下一步去哪」        | ⛔ 未动——本文 §4.4/§4.5    |
| L0 信任层（完工度信号） | 假通知、demo 预填、直出 ID 列——用户第一眼就判「这是个工程半成品」 | ⛔ 未动——本文 §5 U0        |

三个硬数字说明 L2/L3 的严重度：

- 侧栏 **10 个顶级菜单组 / 28 个可见叶子页**；而后端 handoff §5.2 自己推荐的菜单树只有 **6 个顶级组**——前端比后端建议的还要「资源化」；
- 「给张三授权」的核心表单（角色绑定）有 **11 个字段**：主体类型 / 主体 / 角色 / Scope / Scope 组织 / Scope 活动 / 资源类型 / 资源 ID / 任期起止 / 备注——每个字段都是一个后端概念（`org-hr/role-bindings/form.vue`）；
- 统一列表范式 `SrvfListPage` 建好后只推广了 **1/29** 个列表页，其余 28 页各写各的空态/加载/筛选行为。

> 上轮审计 §6 判定「骨架和内脏是健康的」——依然成立。轴模型、权限门控、状态机、留痕都不动；本文全部改动发生在「组织方式」和「表达方式」层。

## 3. 系统性诊断：六条真实任务走查

> 方法：以「一个没读过文档的干部」为镜头，逐步走代码级路径，数三件事——**跨页数、弹窗/字段数、必须预先理解的概念数（概念墙）**。

### W1 办一场活动（活动负责人）

**现路径**：活动 → 活动列表 → 「新建」弹窗（**13 个字段**：标题/类型/承办组织/起止时间/地点/名额/性别限制/报名截止/公开报名/要求保险/两段说明）→ 回列表找到它 → 「管理」进作战室 → 头部「发布」→ 报名 tab 逐条审核 → 活动结束后考勤 tab「提交考勤单据」（批量角色/出勤状态/贡献值/签到签退/服务时长…）→ 一级通过 → 终审通过。
**概念墙（5 面）**：① 新建≠发布（状态机不可见）；② 报名审核不在任何菜单里，藏在作战室 tab；③ 「考勤单」vs「考勤记录」；④ 一级审/终审两级链（按钮出没全靠状态暗示）；⑤ 终审权从哪来（见 W3）。
**大厂对照**：这是典型「一条业务流」，钉钉式做法是在详情页头挂**生命周期条**（草稿→已发布→进行中→已结束→已归档），当前步骤下挂当步动作，下一步永远可见。

### W2 新人从报名到正式队员（招募负责人）

**现路径**：招新轮次 → 新建轮次 → 「管理」进招新作战室（OCR 核验/标门槛/人工通过/评定/公示/一键发号预检/一键发号）→ **人消失了**（变成队员列表里的志愿者，界面没有交代）→ 切到「入队轮次」→ 新建 → 入队作战室（标考核/评估通过/一键入队选部门）→ 正式队员。
**概念墙（4 面）**：① 两道门 funnel（PageIntro 已解释，✅ 上轮成果）；② 「发号」= 建档+发队员编号；③ 发号后的人在哪、下一道门怎么接上——零可视化；④ 同一件事三个词：招新叫「评定」、入队叫「标考核」+「评估」。
**大厂对照**：招聘/CRM 系统的标准答案是 **funnel 看板**——一页看到两道门全链路各阶段人数，点数字下钻。招新作战室的五组 stats 已经是半成品看板，缺的是把两道门连起来。

### W3 任命张三为中队长并让他能终审考勤（权限管理员）——**最难的一条**

**现路径**：组织架构 → 找到中队 → 行操作「更多」→「在任职务」抽屉 → 任命弹窗（**7 字段**，前提：职务定义里已有「中队长」且职务规则允许）→ 这只解决了「职务」；要能终审还得：系统管理 → 角色绑定 → 新建绑定（**11 字段**：主体类型选 POSITION_ASSIGNMENT、主体选刚才那条任职、角色选 attendance-final-reviewer、Scope 选组织并选中队…）→ 还得知道 scoped 判权目前只对「考勤终审 + 活动/报名/考勤点动作」生效。
**概念墙（7 面）**：职务定义 / 任职 / 角色 / 绑定 / 主体类型 / Scope / scoped 生效边界。**这条链没有任何一个界面入口能从头带你走完**；普通人不可能自行发现「任职」和「角色绑定」要各做一次。
**大厂对照**：企微给管理员分权是一个**向导**：「选人 → 选管理范围 → 勾选可以做什么」，系统在背后翻译成权限模型。后端的 `position-assignments/preview` 和 `role-bindings/preview` 预检端点为向导化准备好了防错能力（§4.5-D）。

### W4 发一条全员紧急通知（值班管理员）

**现路径**：通知中心 → 通知管理 → 新建（标题/类型/可见性/可见部门/正文/渠道/置顶）→ 保存成草稿 → 回列表找到 → 「发布」→ 若要短信，再点行操作「发送短信」（计费二次确认，✅ 做得对）。
**概念墙（2 面）**：① 渠道勾选≠实际发送（勾「短信」只是声明可兜底）；② 发布后微信是「机会式推送仅订阅者」。上轮已加解释文案，但模型本身仍绕。
**另一个致命伤**：发完通知，管理员自己的铃铛**永远不会响**——铃铛是假数据，全系统没有任何地方能确认「队员收到了没」（投递明细是已登记的后端缺口）。

### W5 查王五今年的出勤和贡献值（队部管理员）

**现路径**：Ctrl/Cmd+K 全局搜索「王五」→ 队员档案 → 在 **11 个平铺 tab** 里找到「考勤记录」「贡献值」。
**评价**：这条已经是全站最顺的路径（#79/#80 全局搜索是标杆），残余问题：11 个 tab 无分组、默认落「证书」tab 而不是最常用的档案/概览。

### W6 新管理员的第一天（冷启动）

**现路径**：登录框预填 `admin` / `pure-admin`（`src/views/login/index.vue:42-43`）→ 落地首页（6 张导览卡）或工作台 → 铃铛里是「小铭 评论了你」（`src/layout/components/lay-notice/data.ts:31`）→ 想建活动，发现「活动类型」下拉是空的——因为要先去 基础数据→字典 配 `activity_type`，而没有任何界面告诉你这个前置依赖；职务任命依赖职务定义+职务规则，贡献值落分依赖贡献值规则，同理。
**概念墙**：字典 code / 职务 code / 层级权重 / 阈值三件套……**全部是「配置员」概念，却和日常页面混在同一张菜单里**。
**大厂对照**：SaaS 后台的标准答案是**初始化检查清单**（setup checklist）：工作台挂一张「开通进度」卡，逐项打勾（配好字典→建组织→定职务→…），没配好的业务页给「去配置」空态而不是空下拉。

## 4. 升级方案

### 4.1 设计原则（七条军规）

以后每个 PR 的 UI 变更用这七条验收：

1. **任务优先**：高频任务从工作台 ≤3 次点击可达；每个任务链有且只有一条主叙事路径。
2. **说队里的话**：界面词汇 ∈ 术语总表（§4.6）；新名词先进表再上界面；后端码/ID/字段名不作为主要展示。
3. **登录即待办**：第一屏回答「今天我要处理什么」，数字可点、点了直达。
4. **页面自解释**：每页回答三问——这是什么 / 我能干什么 / 下一步去哪（PageIntro + 空态 CTA + 流程条）。
5. **配置藏后面**：干部日常面与管理员配置面物理分区；配置缺失时业务页给「去配置」指引。
6. **危险有礼**：危险操作说后果、分级确认；有 preview 端点的操作先预演再提交。
7. **一处一真相**：同一事实只有一个入口叙事；确需多入口时，次入口标注「同一份数据」。

### 4.2 人物画像草案（待 §8 拍板）

后端与两仓文档**均未定义任何岗位画像**（核查：秘书处/中队长/干部/队委出现次数=0），只有 RBAC 角色→能力映射。这是「谁都不知道这后台为谁而做」的根源。以蓝图 §4 的角色闭环为底，提出六个 persona 草案：

| Persona（草案名） | 岗位直觉        | 高频任务                               | 对应 RBAC 现实                                       |
| ----------------- | --------------- | -------------------------------------- | ---------------------------------------------------- |
| 值班管理员        | 队部值班/秘书处 | 清待办：审报名、一级审考勤、转办终审   | biz-admin                                            |
| 活动负责人        | 各中队干部      | 建活动→发布→审报名→交考勤单            | biz-admin 或 org-admin@本队（scoped 边界见 §7 红线） |
| 人事干部          | 组织/人事负责人 | 挂靠/转隶/任命/免职/归属体检           | ops-admin 或 org-admin                               |
| 招募负责人        | 招新组          | 两道门全流程                           | biz-admin（招新域码）                                |

     向导只是「翻译器」，底层仍是现有两页的同一批端点；原表单保留为设置中心里的「高级模式」。scoped 生效边界文案红线见 §7。

## 7. 红线与边界（升级中不许破坏的）

- **轴模型不破坏**：报名/考勤/证书永不升顶级菜单；funnel/生命周期条是「画出既有状态机」，不是造新流程；
- **不臆造后端**：字段/枚举/权限码/统计一律 live `/api/docs-json`；缺的登 gap-ledger 停下（§5 表）；
- **scoped 文案红线**（后端 handoff 原话背书）：授权向导与角色绑定页必须明示「范围化授权当前仅对 考勤终审 + 活动/报名/考勤点动作 生效，其余业务面仍按全局角色判权」——不许让用户误信 scoped 已全面生效；四个派生角色**不做**全局手工绑定引导；
- **harness 事实**：`src/layout/**`、`src/style/**`、login 等 auth 五文件是 ❌/deny——本文涉及它们的项（铃铛、登录预填、面包屑）全部标注「人工单独 PR」，AI 不绕行；
- **零新依赖**：四件套用 el-steps/el-empty/echarts 现有件；需要 qrcode/cropper/富文本的想法继续走人工逐项决策；
- **不做的**（反过度工程）：按 persona 动态换肤/可配置首页、前端自建审批流引擎、移动端适配专项（小程序是另一条产品线）、把五张人事总表并成 tab（上轮 C 档已否，理由成立）。

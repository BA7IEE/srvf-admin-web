# SRVF Admin 前端 UX 系统性审计（2026-07-06）

> **性质**：可用性 / 信息架构诊断报告。回应主诉「后台很多地方非常奇怪、反常识、很难上手，跟平时常见的后台不太一样」。
> **方法**：4 路分域代码审查（队员+组织人事 / 活动+工作台+招募 / 系统+内容+通知 / 全局横切与 vue-pure-admin 完整版对照）+ 本仓 dev（:8849）实机走查截图取证 + 后端交接文档（`../srvf-nest-api/docs/handoff/admin-web.md`）与蓝图（`srvf-admin-vnext-blueprint.md`）对读。
> **基线**：main `5f67d6e`（PR #60 后）· 后端 v0.37.0。
> **定位**：本报告是**体验层问题单一来源**；结构级功能覆盖见蓝图（105/105 契约全绿，覆盖无缺口）。修复进度在 §7 逐项打勾。

---

## 0. 一句话结论

**功能覆盖完整、契约对账全绿，但这是「给后端契约做的 UI」，不是「给管理员做的产品」。**「反常识感」有三条根因，全部可修，且大多不需要动骨架：

1. **契约覆盖驱动铺页**——过去的主线是把 67 条未消费后端路径逐一补齐（PR #36–59），验收是「接口通了/对账全绿」而非「角色闭环好用」。后端每个资源族都长成了一个菜单项，页面是端点的镜像，不是任务的容器。
2. **后端概念漏进用户界面**——权限码、状态枚举、字典 code、scope 类型、过滤表达式这些应该被前端「翻译」掉的东西，大量原样印在界面上。UI 在向用户解说自己的实现。
3. **starter 底座未清算**——登录落地页还是脚手架时代的占位页，浏览器标题还是 `pure-admin-thin-max-ts`，侧栏挂着 demo「异常页面」组。

**骨架和内脏是健康的（轴模型、权限门控、状态机、留痕），病在皮肤和五官——而皮肤和五官正是用户唯一能看到的部分。**

## 6. 有意为之、不应「修」的部分

- **轴模型本身**（报名/考勤/证书长在活动/队员详情内，不做顶级菜单+手选父级）——后端 handoff §1 钦定反模式条款，是本系统比通用 CRUD 后台更贴业务之处。修**可达性补偿**，不修模型。
- **工作台跨轴横扫 + 状态预置下钻**——方向正确，已落地。
- **按钮级权限门控、终审专用错误码、软删留痕、短信计费二次确认**——「重」的地方恰是做得扎实的地方。
- **菜单=前端静态+权限过滤**（无 asyncRoutes/getMenuList）——双方架构约定，非缺陷。

## 附 · 与常见后台差异五句话

1. 登录落地占位页而非工作台（常见后台落 dashboard/待办）；
2. 界面说「后端话」（权限码/枚举/表达式直出，常见后台全量翻译）；
3. 列表页搜索/批量/列设置等标配缺席或不统一；
4. 菜单是端点分布图而非工作分区图（权限 4 入口、人事 6 平行表）;
5. 品牌红吞掉了危险语义（全按钮一片红）。

# UI / UX 规则

## 一、总体原则

- 页面按任务设计，不按数据库资源堆菜单。
- 子资源尽量归入父级详情页，避免“先选择一个 X 才能管理 Y”的顶级菜单反模式。
- 保持 Pure Admin / Element Plus 的后台一致性。
- 参考包只看布局和交互，不复制业务模型。

## 二、后端 admin-web 轴模型

- 活动详情页：报名、考勤。
- 队员 360：档案、证书、紧急联系人、保险、组织归属、任职、参与记录。
- 组织架构：组织树、组织成员、在任职务。
- 系统管理：用户、角色、权限诊断、职务、规则、角色绑定、分管、字典、贡献、附件、审计、短信、设置。

## 三、权限交互

- 页面和按钮由真实权限码控制。
- 无权限时可隐藏或禁用，具体以后端 action-state 语义为准。
- 不得用 mock 权限码。

## 四、视觉质量

- 业务页优先保证表格、筛选、弹窗、Drawer、详情页一致。
- 不为视觉优化破坏主链。
- UI 重构必须排在 P0 主链风险修复之后。
